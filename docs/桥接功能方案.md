# 基于 Moquette (0.18.0) 的内嵌式 MQTT Broker 跨服务桥接方案设计书

本方案旨在不破坏 Moquette 核心源码的前提下，通过外挂**事件拦截器**与**独立路由引擎**，为内置的 Moquette Broker 扩展高可用、高性能的跨 Broker 双向桥接功能，并配套提供完整的可视化 Web 管理后台。

---

## 一、 系统架构设计

桥接引擎作为独立模块嵌入在 Moquette 进程中。它对内作为 Moquette 的拦截插件，对外作为一个标准的 MQTT 客户端（基于 Eclipse Paho 库）连接远程目标。

### 1. 核心组件定义

* **本地 Broker（Moquette）**：承载本地物联网设备与客户端的连接。
* **桥接路由引擎（Bridge Engine）**：负责跨 Broker 的数据路由、主题映射、流量削峰与防死循环控制。
* **远程 Broker**：任何标准的 MQTT 服务端（如 EMQX、Mosquitto 或云端物联网平台）。

### 2. 双向数据流向拓扑

```
[ 本地物理设备 ]
       │ (PUBLISH)
       ▼
┌────────────────────────────────────────────────────────┐
│ 本地 Moquette Broker                                   │
│   │                                                    │
│   ▼ (事件拦截器捕获)                                     │
│ ┌────────────────────────────────────────────────────┐ │
│ │ 桥接路由引擎 (Bridge Engine)                        │ │
│ │                                                    │ │
│ │   ┌───────────────┐        ┌───────────────────┐   │ │
│ │   │ 出向异步队列  │        │ 入向标签注入机制  │   │ │
│ │   └───────┬───────┘        └─────────▲─────────┘   │ │
│ └───  ──────┼──────────────────────────┼─────────────┘ │
└─────────────┼──────────────────────────┼───────────────┘
              ▼ (Paho Client 转发)       ▲ (Paho Client 订阅)
       ┌─────────────────────────────────┴┐
       │         远程 MQTT Broker         │
       └──────────────────────────────────┘

```

## 二、 核心数据流控制设计

### 1. 出方向流控（Outbound: 本地 → 远程）

1. **事件捕获**：本地设备发布消息至本地 Moquette，触发 `InterceptHandler` 的 `onPublish` 同步回调。
2. **动态过滤**：路由引擎检索该链路的动态路由规则。若主题匹配成功则放行，否则立即结束拦截。
3. **线程隔离**：匹配成功的消息剥离 Netty 字节流，将元数据（Topic, Payload, QoS, Retain）投递至内部的轻量级无锁环形队列（如 Disruptor ）。
4. **异步外发**：独立的工作线程池（Bridge Worker Pool）从队列中消费消息，驱动远程 MQTT 客户端发送至远程 Broker。

### 2. 入方向流控（Inbound: 远程 → 本地）

1. **远程订阅**：桥接客户端启动时，根据入向规则向远程 Broker 订阅指定主题。
2. **异步接收**：远程消息到达客户端回调接口。
3. **防环路由注入**：引擎调用 Moquette 的 `Server.internalPublish()` 接口。注入时**必须**显式指定一个特定的客户端标识（如 `BRIDGE_INBOUND_AGENT`）。
4. **本地分发**：本地 Moquette 将消息无缝推送到订阅了该主题的本地设备。

## 三、 生产级高可用与防御设计

### 1. 消息环路死循环阻断（Loop Prevention）

* **风险**：若双向桥接的主题存在交集（例如两端均同步 `telemetry/#`），消息会在两个 Broker 间无限循环转发，导致系统雪崩。
* **设计**：出方向拦截器在捕获消息时，优先校验消息的 `ClientID`。**一旦发现 `ClientID == BRIDGE_INBOUND_AGENT`，直接丢弃该事件，阻断其再次外发。**

### 2. 线程与内存保护

* **Netty 隔离**：严禁在拦截器回调中直接执行网络 I/O。拦截器仅做“匹配与投递队列”两件事，耗时控制在微秒级，确保 Moquette 本地网络吞吐不受远程网络波动影响。
* **背压与缓冲区控制**：每条桥接链路分配独立的内存缓冲区上限。当远程断网导致队列积压满时，触发**背压策略**（默认选择“丢弃最老消息”），严防内存溢出（OOM）。

### 3. 断线弹性退避（Resilience）

* 远程客户端必须开启自动重连，但重连间隔需采用**指数退避算法**（如：1s, 2s, 4s, 8s... 最高上限 60s），避免远程 Broker 宕机时，本地引擎高频重试耗尽 CPU 资源。

## 四、 可视化 Web 管理后台设计

为实现桥接链路的可视化运维与动态配置，配套设计以下前端页面：

### 1. 桥接链路总览页（List & Dashboard）

* **顶部指标卡**：
* **健康状态看板**：以图形化（绿/黄/红）展示当前“已连接 / 重连中 / 已断开”的链路数量。
* **实时速率仪表盘**：全双工展示当前整个 Broker 的出向、入向消息吞吐速率（msg/s、KB/s）。


* **全功能数据表格**：
* 展示内容包括：链路别名、远程 IP 地址、当前连接状态、运行方向、累计发送/接收消息总量。
* **动态控制栏**：提供一键启用/禁用的 **Switch 开关**（实时断开/启动底层客户端）；提供独立日志弹窗链接，用于查看特定链路的握手或认证错误日志。



### 2. 链路配置表单页（Configuration Form）

* **基础连接区块**：
* 输入项：链路别名、远程 URL（带格式校验）、Client ID、Keep Alive 间隔。
* 安全认证：单选框切换（匿名 / 账号密码 / SSL证书）。选择证书时，展开标准的 CA 证书、客户端证书上传组件。


* **高级动态路由规则组件**：
* 提供动态加行表格，每行包含：方向（出向/入向）、源主题（支持通配符 `+`、`#`）、目的主题（支持前缀重写或变量提取如 `${1}`）、期望 QoS（透传/强制转换）、Retain 处理机制。
* **安全提示预警**：当前端检测到用户在同一链路上配置了相同或重叠的双向规则时，表单底部动态弹出高亮警告，提示检查防环路配置。


* **连接测试功能（Test Connection）**：
* 表单下方常驻“测试连接”按钮。点击后，后端通过临时线程进行探测性握手，前端根据返回结果即时提示“连接成功”或“连接超时/认证失败”的详细堆栈，避免无效保存。

## 五、 后端 RESTful 接口（API）契约设计

桥接引擎在进程内通过轻量级 Web 框架暴露以下接口，供前端后台调用：

| 请求方法 | 接口路径 | 描述 | 主要数据流向 |
| --- | --- | --- | --- |
| `GET` | `/api/bridges` | 获取所有桥接链路的配置信息与当前实时状态 | 后端存储 $\rightarrow$ 前端列表展示 |
| `GET` | `/api/bridges/{id}/stats` | 通过长轮询/WebSocket 实时获取指定链路的吞吐量与速率 | 引擎监控计数器 $\rightarrow$ 前端图表 |
| `POST` | `/api/bridges` | 提交并动态启动一条全新的桥接链路 | 前端表单 $\rightarrow$ 引擎动态实例化 |
| `PUT` | `/api/bridges/{id}` | 修改已有桥接配置。后端将平滑断开旧连接、应用新配置并重启实例 | 前端表单 $\rightarrow$ 引擎热重载 |
| `POST` | `/api/bridges/{id}/toggle` | 接收 `enable=true/false`，无缝在线启停特定链路 | 前端开关 $\rightarrow$ 引擎连接控制 |
| `DELETE` | `/api/bridges/{id}` | 从系统内彻底移除某条桥接链路配置（仅限断开状态） | 前端请求 $\rightarrow$ 后端销毁实例 |
| `POST` | `/api/bridges/test-connect` | 接收临时表单数据，发起一次性连接探测 | 前端轻量请求 $\rightarrow$ 临时网络握手 |
